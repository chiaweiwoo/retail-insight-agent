Structural Weaknesses (Code-Verified)

Problem 1

LLM used to compute a number

sales_deviation_agent wraps a single sales_deviation_calculator Python function inside create_tool_calling_agent + AgentExecutor. The LLM is asked to call one tool and return a float. This adds latency, cost, and non-determinism to a formula: (actual − forecast) / forecast × 100. If the model misformats the output, result.get("output") silently returns the wrong value.

Redesign Fix

Move signal computation offline to a precompute step (rca signal). Run once as a deterministic pandas/SQL job, persist to a signals table. Never recompute at agent runtime.

Problem 2

No planner — all agents always run

route_deviation returns either ["__end__"] or ["traffic","sales","inventory","promotions","news"]. Every deviation triggers all 5 agents at equal cost, regardless of signal type. A holiday lift that is obviously promotions-driven still runs the inventory and news agents at full token cost.

Redesign Fix

Add a Planner node that reads the signal, calendar context, and prior memory to select which agents to run this round. A clear holiday lift may only need calendar + promotions. A supply disruption needs inventory + news. Cost scales with complexity, not uniformly.

Problem 3

Critique loop never gains new information

The manager → critique loop runs up to 5 iterations. Each pass receives the same five tool summaries and rewrites prose based on style feedback (baseline values, driver order, sentence clarity). No new Snowflake queries are made. At iteration 5, you have spent 5× LLM cost and produced a better-worded version of the same analysis. The critique checks formatting compliance, not evidence gaps.

Redesign Fix

Replace the style-review loop with an evidence-gap critic. Critic outputs typed gaps (missing_internal_evidence, baseline_conflict, missing_external_context, unavailable_data) and a continue_investigation flag. When true, the next round calls new tools and gathers evidence the previous round didn't have. The loop grows the evidence base, not the prose.

Problem 4

No evaluation means no measurable quality

observability.py tracks token counts and latency per agent — useful for cost management. There is no mechanism to score output quality. You cannot answer: "Is today's report better than yesterday's? Did the prompt change improve anything? Which locations have low-confidence RCAs?"

Redesign Fix

Add a two-layer evaluation node: (1) 8 deterministic rule checks (no currency terms, no product root cause, evidence non-empty, etc.) as cheap fast gates; (2) LLM alignment judge that scores 0.0–1.0 against guardrails. Store scores per run in a queryable table. Every prompt change can be A/B tested against a consistent baseline.

Problem 5

Monolithic prompts doing five jobs at once

The manager prompt is ~400 lines encoding: output format, baseline compliance rules, contradiction avoidance, zero-sales protocol, driver prioritization, JSON formatting, domain knowledge, and iteration tracking. The critique prompt is ~500 lines doing the same. Any model update that handles the prompt differently silently breaks the output structure.

Redesign Fix

Narrow mandate per node: Planner (~10 lines, select agents), Critic (~20 lines, return gaps JSON), Coordinator (~30 lines, synthesise brief). Evaluation logic lives in Python, not prompts. Each prompt is independently testable.

Interview Talking Points

Opening narrative

"In the first version, we got multi-agent orchestration running end-to-end — LangGraph, Snowflake, Streamlit, all connected. But in production we noticed that the critique loop was polishing prose rather than gathering evidence, every deviation triggered all five agents at the same cost regardless of signal type, and there was no quality score so we couldn't tell if a prompt change helped or hurt. In the redesign I addressed each problem with a specific engineering decision: precomputed signals, a planner that selects agents dynamically, a gap-type critic that drives real investigation rounds, and a two-layer evaluation system — deterministic checks first, LLM judge second — so every run produces a score you can track over time."

On the critique loop

"The original critique ran up to five iterations over the same data. That's prompt engineering masquerading as intelligence. The LLM was checking formatting compliance — baseline values present, driver order correct, no contradictions. Valid checks, but not new information. I redesigned the loop so the critic identifies evidence gaps with types — missing_internal_evidence, baseline_conflict, unavailable_data — and those gap types tell the planner what to do next round. Now each iteration actually calls new tools."

On evaluation

"I'd build the scoring harness first next time, then build the agents. Without a score you can track per run, you're flying blind every time you change a prompt. Deterministic checks are cheap and reliable — eight rule-based gates catch the obvious violations. The LLM judge then adds calibration and nuance scoring on top. You need both layers."

On model routing

"Running everything on the same model was leaving quality on the table in one place and wasting cost in the other. The planner and critic need consistent JSON output, not deep reasoning — that's the fast model's job. The coordinator synthesises across all evidence and needs better nuanced reasoning — that's the deep model. Match model capability to task complexity."

Structural Weaknesses (Code-Verified)

Problem 1

All 5 LLM calls are sequential with no early-exit optimisation

run_llm_analysis in core/pipeline.py is a linear method chain: identify_process → analyze_thread → next_responders → scenario_match → extract_rl_id → draft_email. There is one early exit (if is_rtv=False, return early), but calls #2 and #3 could run in parallel since they only need the raw email data. The current design forces them to run one after another, increasing end-to-end latency unnecessarily.

Redesign Fix

Use a LangGraph parallel branch: analyze_thread and next_responders read only email data and can fire simultaneously. The scenario_match waits on both. An asyncio-based graph cuts total latency on the critical path by ~30–40%.

Problem 2

Scenario matching returns an integer index into a spreadsheet

extract_scenario_insights asks the LLM to return an index field, then does df_scenario.loc[idx]. This means the LLM is matching an email to a row position in a SharePoint Excel sheet. If anyone inserts or reorders rows in the spreadsheet, all prior index matches are wrong. The LLM is also not designed to reason about positional indices — it is reasoning about semantics but forced to express that as a number.

Redesign Fix

Embed scenario descriptions as vectors at startup. At runtime, do semantic cosine similarity search over the vector store to find top-3 candidate scenarios. Pass those candidates to the LLM to select the best match. Decouples the match from spreadsheet row order entirely. Use pgvector on Supabase or a lightweight FAISS index.

Problem 3

LLM used as a string matcher for RL IDs

RTVFollowupTools.get_llm_best_match in rtv/tools.py sends fuzzy-matched RL ID candidates to the LLM to pick the best one. This adds a full LLM API call latency and cost for a task that is fundamentally a string comparison — one that good deterministic fuzzy matching (already in place with fuzzywuzzy) could handle. The LLM adds variance, not accuracy, to a high-confidence fuzzy match.

Redesign Fix

Use deterministic matching only: exact match → normalised token sort → Levenshtein distance with a threshold (e.g. ≥0.92). If no match above threshold, flag as rl_id_uncertain: true in the output and include it in the draft's TO-DO section rather than asking the LLM to guess.

Problem 4

JSON parsing via regex is fragile

Every LLM call in rtv/llm.py parses the response with re.search(r'\{.*\}', result, re.DOTALL)[0]. If the model wraps the JSON in a markdown code fence (```json ... ```), adds a preamble sentence, or returns nested objects, the regex either fails silently, extracts the wrong slice, or raises an unhandled exception. There are no structured output guarantees.

Redesign Fix

Use response_format={"type": "json_object"} (GPT-4o+) or Pydantic structured outputs with .with_structured_output(ScenarioMatch). The model is forced to return valid JSON matching the schema. Remove all regex parsing. If the model cannot conform, the API raises a typed error — not a silent wrong parse.

Problem 5

Evaluation is offline and manual — not inline

The evaluation system (notebooks 09, 22, 25, 26, etc.) is a separate offline process. A human reviewer marks a draft as "Done" in the Snowflake table (FEEDBACK_STATUS = 'Done'), and only then can the accuracy calculation run. There is no inline quality signal for each new email processed. A failing prompt can process hundreds of emails before the offline evaluation catches the degradation.

Redesign Fix

Add an inline evaluation node to the pipeline: after every draft is generated, an LLM judge verifies that (1) scenario matches the email thread, (2) draft addresses all required follow-up actions, (3) recipients are appropriate, (4) no confidential information is leaked. Score stored per email in Snowflake. A degrading score triggers an alert before the queue backlog fills.

Problem 6

No structured workflow graph — just a class method chain

Pipeline.run_llm_analysis is a flat imperative Python method with nested if/else for early exits. There is no LangGraph graph for the email processing flow itself. This means: no conditional parallel branches, no reusable node definitions, no typed state transitions, and no way to add retry/fallback logic declaratively. The flow is opaque — you can only understand it by reading the method top to bottom.

Redesign Fix

Define the email pipeline as a proper LangGraph StateGraph: each step is a named node, routing is via conditional edges, parallel analysis via fan_out edges. State is a typed EmailState TypedDict. Graph structure is inspectable, testable node-by-node, and extendable without touching unrelated nodes.

Interview Talking Points

Opening narrative

"The email automation project shipped a working pipeline that processed 2,500 cases per month and cut manual handling from 80 to 20 hours. But looking at the code, there are structural gaps that would matter at higher scale or with a stricter quality bar. The scenario matching returns an integer row index into a SharePoint spreadsheet — that breaks silently if someone inserts a row. JSON parsing uses regex on free-text LLM output — that fails unpredictably as models change. And evaluation was offline and manual, which means a failing prompt can degrade hundreds of drafts before anyone notices. If I were redesigning it, I'd fix those three things first: vector-based scenario matching, structured outputs everywhere, and an inline evaluation node on every draft."

On scenario matching

"The LLM is asked to return an integer index into a spreadsheet row. That conflates semantic reasoning with positional lookup. The model reasons about meaning but expresses it as a number — and the number is meaningless if the spreadsheet changes. The fix is to separate concerns: embed scenario descriptions once into a vector store, do cosine similarity at runtime to get the top-3 candidates, then let the LLM pick the best match from three readable options. Now the spreadsheet can be reordered without breaking anything."

On inline evaluation

"The evaluation setup in this project is actually more sophisticated than most — there's a ground truth labelling system, accuracy tracking per scenario, a retest workflow. The gap is that it's all offline. The production queue can process a hundred emails on a degraded prompt before the weekly review catches it. Adding an inline judge node that scores every draft and writes to a Snowflake quality table gives you a real-time signal. You don't need human review to go faster — you need a machine proxy that alerts you when quality drops, so humans can review the right cases."

On the LLM as string matcher

"Using an LLM to pick the best fuzzy match for a 10-character RL ID number is like using a chef to wash dishes. The LLM adds latency and variance to a task that deterministic string similarity handles better. I'd remove that LLM call entirely and instead surface the uncertainty explicitly — if no match clears the threshold, mark the RL ID as uncertain in the output and include it as a to-do in the draft. That's more honest and faster than asking the LLM to guess."

If asked about the AI evaluation & prompt governance bullet on your résumé

"We built a layered evaluation approach: first, ground truth labelling where human reviewers marked draft outcomes as passed/failed across scenario types. Second, an accuracy tracking system per scenario stored in Snowflake, so we could see which scenario types had low AI effectiveness and needed prompt work. Third, automated retest — when a user flagged a bad draft, we'd retrigger the pipeline and compare old vs. new output. What I'd add if building from scratch is an inline scoring node so the system catches regressions in real-time rather than waiting for human review."
